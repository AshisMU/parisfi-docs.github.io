Pipelines & Workspaces

Workspaces allow you to chain multiple automated steps together into a seamless, intelligent document pipeline — including conditional routing and per-item loops.

Workspaces vs Orchestration Pipelines: Workspaces are designed for fast, linear processing of single documents (e.g. Auto-Categorise → Extract → Branch). If you need to combine multiple documents, generate PDFs/Word docs from templates, or insert human-in-the-loop review gates, use Orchestration Pipelines instead.

Step Types

A Workspace is built out of sequential Steps. Each step can be one of four types:

  • Auto-Categorize: Instantly tags the document using the system taxonomy. Runs before all other steps when enabled.
  • Analysis: Runs an LLM prompt from your library against the document.
  • Agent Tool: Executes a specialized extraction tool with an optional JSON schema, template, and guardrails.
  • Branch 🔀: Routes the pipeline to a YES path or a NO path based on a condition — no code required.
  • Repeat 🔁: Runs an Agent Tool once for each item in a list extracted by a previous step.

Building a Pipeline

Steps are added and reordered visually in the Workspace editor:

  1. Click + New Workspace and give it a name.
  2. Use the toolbar buttons — Analysis, Agent Tool, Branch, Repeat — to add steps.
  3. Use the ▲ ▼ arrows on each step card to reorder. Order matters: Branch and Repeat steps must come after the steps whose output they reference.
  4. Click the ✎ pencil icon on a Branch or Repeat card to re-open the configuration wizard and edit it.
  5. Click Save Pipeline when done.

Branch Step — Conditional Routing

A Branch step evaluates a condition and routes the remaining pipeline to one of two paths:

  • YES path: steps that run when the condition is true.
  • NO path: steps that run when the condition is false.

The Branch wizard lets you build conditions in plain language — no code needed:

  • Document category is / contains … — route based on Auto-Categorisation result
  • Extracted field from a prior step equals / contains … — route based on what a tool found

An Advanced toggle reveals the underlying Jinja2 expression for power users who want full control.

Safe by default: If the condition cannot be evaluated (for example, the referenced field is missing or the prior step hasn't run), the Branch defaults to the NO path and the pipeline continues without error.

Repeat Step — Per-Item Loops

A Repeat step takes a list produced by a previous Agent Tool and runs another tool once for each item in that list. This is ideal for:

  • Running a KYC check on every director extracted from a corporate filing
  • Scoring each line item in an invoice against a compliance rule
  • Generating a report card for each candidate listed in a shortlist

The Repeat wizard automatically discovers array fields from the prior step's JSON Schema, so you never need to write an expression manually. An Advanced toggle exposes the iterate_over expression for custom use.

Inside the looping tool's prompt, reference the current item with {{current_item}} (or a custom loop variable name you define).

Safe by default: If the list resolves to something that isn't a list (null, string, etc.), the Repeat step runs 0 iterations and the pipeline continues without error.

Pipeline Flow Diagram

When a pipeline contains Branch or Repeat steps, a read-only Pipeline Flow diagram automatically appears below the step list in the editor. This gives you a clear visual overview of the pipeline topology before saving.

Input & Target Source Logic

For Analysis and Agent Tool steps, you can override the Target Source — the data the step processes:

  • Original Document (Default): process the document's OCR content directly.
  • Previous Step Output: feed the text output of the preceding step as this step's input.
  • Named Step: reference any prior step by name for chained processing.

Variable Chaining

Use {{Step Name.output}} in a step's variable value to inject the full text output of a preceding step. The schema-aware variable helper popup makes this easy — it shows individual field names from the tool's JSON Schema so you can reference exact fields without typing.

Automated Re-aggregation

When you run a workspace on a document within a collection, Parsifi automatically triggers a re-aggregation of the collection's results, ensuring your project-level dashboard is always up to date.

Database Storage & Resiliency

All Workspace definitions are stored securely in the Parsifi application SQLite database, keeping your data entirely local and protected from accidental flat-file deletion. If migrating datasets, an idempotent importer safely attempts to restore pipelines from legacy `workspaces.json.bak` configuration backups.

Reliability Engine: Every Workspace execution is backed by Parsifi's Reliability Engine, featuring automated retries and JSON repair for maximum data integrity.
