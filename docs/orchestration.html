Orchestration Pipelines

Orchestration Pipelines take document automation beyond linear workspaces, enabling multi-input, multi-output workflows with interactive visual canvas editing, human-in-the-loop review gates, and automatic variable mapping.

Workspaces vs Pipelines

While Workspaces are perfect for simple, linear processing of single documents (e.g. OCR → Extract → Categorise), Orchestration Pipelines are designed for complex, multi-document scenarios:

Feature Workspaces Orchestration Pipelines
Input Single Document Multiple Inputs (Documents, Text, URLs)
Topology Linear (with basic branching) Directed Acyclic Graph (DAG)
Output JSON / Meta Data Generated Documents (PDF, DOCX)
Data Mapping Manual JSON Pathing Automatic via Variable Register
Human Review None Native Review Gates
Data Traceability Not available Visual Data Lineage Map 🗺
Edit Safety Overwrites immediately Full Version History & Restore

Input Slots

Unlike workspaces that process a single implicit document, pipelines define explicit Input Slots. A pipeline will wait until all required input slots are filled before it can be run.

  • Document: E.g. "Primary Contract" or "Identity Passport"
  • Text Section: E.g. "Special Instructions" or "Company Context"
  • URL Link: E.g. "Company Website"

Node Types

A pipeline is composed of nodes on a canvas. There are four types of nodes:

Extractor

Runs an Agent Tool to pull structured data out of documents and into the Variable Register.

Generator

Merges data from the Variable Register into a Template (DOCX/PDF) to output a new file.

Review Gate

Pauses the pipeline. A human must review the extracted data and click Approve or Reject to continue.

Logic Node

Evaluates a condition (e.g. `status == APPROVED`) and halts execution along a path if false.

Building Pipelines: Two Ways

1. The Visual Canvas Editor

For explicit control, the Data Flow Graph provides a full interactive visual editor. Click Edit Canvas to reveal the node palette. You can drag Extractors, Generators, and Review Gates onto the grid, wire them together with click-and-drag edges, and configure their underlying tools and templates right in the inspector panel.

2. The Smart Wizard

If you prefer a guided approach, the Smart Wizard asks you what input slots you need, which agent tools should run, whether you need a human review gate, and what template should be generated. It will automatically construct the underlying pipeline graph for you.

Data Lineage Visualisation: When a pipeline is running or has completed, you can click on any node in the graph to view exactly what data went in and what data came out. This provides complete transparency for auditing.

Data Lineage Map New

The Data Lineage Map is a visual overlay on the pipeline canvas that makes the implicit variable flow completely explicit and verifiable before you run anything. Toggle it on by clicking the 👁 Data Map button in the graph toolbar (visible once a pipeline is loaded).

When active, the overlay injects small variable nodes between your Extractor and Generator nodes, with dashed edges showing which variables are produced and which are consumed:

Mapped ✓

Variable is produced by an Extractor and consumed by a Generator. Data will flow correctly.

Unmet ⚠

A Generator (Template) requires this variable, but no Extractor on the canvas produces it. The output document will have a blank field.

Collision ⚡

Two or more Extractor nodes produce a variable with the same name. One will overwrite the other — resolve by renaming a tool's output field.

The Data Map is read-only and purely visual. Toggle it off at any time to return to the standard execution-order canvas. It does not affect how the pipeline actually runs.

Pipeline Versioning New

Every time you save a pipeline canvas — either via Save Canvas in the editor or when initially creating a pipeline with the Smart Wizard — Parsifi automatically saves an immutable version snapshot of the graph. This means your nodes, edges, and their configurations are never destructively overwritten.

Accessing Version History

Click the 🕐 History button in the graph toolbar to open the Version History panel on the right. It displays each saved version with its version number, the date it was saved, and a short change note (e.g. “Canvas save #3”).

Restoring a Previous Version

Click Restore next to any historical version to load its nodes and edges back onto the canvas. This does not auto-save — you must review the restored graph and click Save Canvas to commit it. Saving after a restore creates a new, higher-numbered version, preserving the full history without losing any state.

Compliance benefit: In regulated industries, the version history provides a complete, timestamped audit log proving exactly which pipeline logic was active when a specific set of documents was processed.
